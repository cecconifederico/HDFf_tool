# Results Structure

This directory contains the outputs and data structures generated by the HFDf estimation pipeline across different vigilance states: **Awake** and **Sleep**, mapped across the neurophysiological datasets (`A1`, `M1`, and `S1`).

## Execution Logic & Variable Output
Depending on the input dataset loaded into the pipeline, the execution script operates in one of two mutually exclusive modes and exports a single primary MATLAB table variable into the workspace binary file (`HFDf_totale_curves.mat` or `HFDf.mat`):

1. **Awake Processing Mode:** If raw wakefulness data is provided, the script outputs the table **`HFDf_awake`** (Matrix size: **47 rows × 7 columns**).
2. **Sleep Processing Mode:** If raw sleep data is provided, the script outputs the table **`HFDf_sleep`** (Matrix size: **101 rows × 7 columns**).

---

## 1. Table Dimensions & Row Segmentation

### A. Awake State Matrix (`HFDf_awake`)
The wakefulness table populates exactly **47 rows**, corresponding to the independent recording channels tracked during the awake condition:
- **A1 (Auditory Cortex):** 10 channels (Rows 1 to 10)
- **M1 (Primary Motor Cortex):** 20 channels (Rows 11 to 30)
- **S1 (Primary Somatosensory Cortex):** 17 channels (Rows 31 to 47)

### B. Sleep State Matrix (`HFDf_sleep`)
The sleep table populates exactly **101 rows**, dynamically stacking the channels across three distinct macro-stages of sleep (**N2**, **N3**, and **REM**) within each anatomical area:

* **A1 (Auditory Cortex) ➔ Total: 17 rows** (Rows 1 to 17)
    * *N2 Stage:* 6 rows
    * *N3 Stage:* 6 rows
    * *REM Stage:* 5 rows
* **M1 (Primary Motor Cortex) ➔ Total: 45 rows** (Rows 18 to 62)
    * *N2 Stage:* 16 rows
    * *N3 Stage:* 16 rows
    * *REM Stage:* 13 rows
* **S1 (Primary Somatosensory Cortex) ➔ Total: 39 rows** (Rows 63 to 101)
    * *N2 Stage:* 14 rows
    * *N3 Stage:* 14 rows
    * *REM Stage:* 11 rows

---

## 2. Output Data Table Schema (Identical for both `HFDf_awake` and `HFDf_sleep`)

Both tables adhere to the exact same 7-column configuration layout:

| Column Name | Data Type | Analytical Description / Meaning | Example |
| :--- | :--- | :--- | :--- |
| **Area** | `string` / `cell` | The specific cortical area of the recording. | `'A1'`, `'M1'`, `'S1'` |
| **Canale** | `string` / `cell` | Alphanumeric unique global identifier of the target channel. | `'GD013Lu_8W'`, `'GD013Lu_8N'` |
| **TipoElettrodo** | `string` / `cell` | Classification extracted from the first 2 letters of the channel name. | `'GD'` |
| **SoggettoID** | `string` / `cell` | Anonymized Subject ID extracted from the 3 central numeric digits. | `'013'` |
| **Elettrodo** | `string` / `cell` | Local physical identification of the electrode (remaining central string before the state). | `'Lu_8'` |
| **HFD_vs_K** | `double` (vector) | Vector array storing the discrete Higuchi Fractal Dimension ($HFD_f$) evaluated dynamically for each channel while scaling $k$ from $k_{min}$ to $k_{max}$. | `[2, 3, 4, ..., 35]` |
| **k_range** | `double` (vector) | The independent scale factor step vector used for the iteration ($k \in [k_{min} \dots k_{max}]$). | `[2, 3, 4, ..., 35]` |

## 3. Automated Channel Label & State Parsing Rule

To systematically decode each raw column header into anatomical, clinical, and behavioral metadata, the pipeline implements a string parsing rule on the **Canale** label.

The first components (Electrode Type, Subject ID, and local Electrode index) follow an identical extraction logic across all conditions, while the **final character** explicitly categorizes the target vigilance state:

* **Characters 1–2 ➔ `TipoElettrodo`:** Type of electrod  (e.g., `GD`).
* **Characters 3–5 ➔ `SoggettoID`:** Unique three-digit numerical code for the experimental subject (e.g., `013`).
* **Remaining Center Sub-string ➔ `Elettrodo`:** Spatial coordinate or location tag of the sensor within the cluster (e.g., `Lu_8`).
* **Final Character ➔ Vigilance State Code (`Stato`):**
    * **`W`** ➔ **Awake (Veglia):** Baseline quiet wakefulness recording. Maps to the `HFDf_awake` variable.
    * **`N`** ➔ **N2 (Non-REM Stage 2):** Light/Intermediate synchronous sleep phase. Maps to the `HFDf_sleep` variable.
    * **`D`** ➔ **N3 (Non-REM Stage 3):** Deep slow-wave sleep phase (Delta sleep). Maps to the `HFDf_sleep` variable.
    * **`R`** ➔ **REM (Rapid Eye Movement):** Desynchronized paradoxical sleep phase. Maps to the `HFDf_sleep` variable.

### Examples of Automated Parsing:
- `GD013Lu_8W` ➔ Type: `GD`, Subject: `013`, Sensor: `Lu_8`, State: **Awake** (Stored in `HFDf_awake`)
- `GD013Lu_8N` ➔ Type: `GD`, Subject: `013`, Sensor: `Lu_8`, State: **N2 Sleep** (Stored in `HFDf_sleep`)
- `GD013Lu_8D` ➔ Type: `GD`, Subject: `013`, Sensor: `Lu_8`, State: **N3 Sleep** (Stored in `HFDf_sleep`)
- `GD013Lu_8R` ➔ Type: `GD`, Subject: `013`, Sensor: `Lu_8`, State: **REM Sleep** (Stored in `HFDf_sleep`)

## 4. Methodological Notes on `HFD_vs_K` Calculation
1. **Segment Median Windowing:** For each individual channel, the raw continuous time-series block (13,600 data points per session) is processed dynamically. To extract a stable trajectory that defines the structural complexity profile of the channel across the scale spectrum, a median aggregation rule is applied over the calculated sub-segments.
2. **Statistical Quality Control:** If a specific channel displays a poor linear regression fit ($R^2 < 0.90$) due to experimental artifacts or high-frequency noise, its `HFD_vs_K` array entries are safely forced to `NaN` placeholders to protect downstream statistical comparisons from anomalous outliers.
